PART 1: PROJECT OVERVIEW & TECHNICAL STACK
-------------------------------------------------------------
Project Name: Smart Expense and Habit Tracker
Architecture / Stack: MERN Stack (MongoDB, Express.js, React.js via Next.js, Node.js)
Purpose: A full-stack web application designed to help users track their daily financial expenses, set monthly budgets, and concurrently monitor daily habits (study, sleep, exercise) using visual analytics and smart alerts.

1. FULL TECHNICAL STACK DETAILS
Frontend (Client-Side):
- Framework: Next.js (Version 16 - React 19)
- Language: JavaScript (JSX)
- Styling: Custom Vanilla CSS for deep customization (Glassmorphism layout, dynamic themes).
- Port Used: 3000 (Default Next.js port: http://localhost:3000)
- Typography: Inter (Google Fonts)
- Iconography: Inline structural Outline SVGs (no heavy icon libraries to maximize performance).
- Charts: Fully Custom Structural SVG Charts (Responsive Donut Chart, Bar Chart with gridlines, and Area-Gradient Sparkline Charts) without relying on heavy external libraries like Chart.js.

Backend (Server-Side):
- Framework: Node.js with Express.js (server.js)
- Language: JavaScript
- Port Used: 5000 (Defined in .env, API runs at http://localhost:5000/api)
- Database: MongoDB (Hosted on MongoDB Atlas cloud) using mongoose ORM. Database name: smart_tracker.
- CORS: Cross-Origin Resource Sharing is enabled allowing localhost:3000 to interact with localhost:5000 securely. 

2. AUTHENTICATION & SECURITY
To handle login, signup, and protected routes, the application uses JSON Web Tokens (JWT).
- Storing Passwords: When a user registers, their password is computationally hashed and salted using the "bcryptjs" library before hitting the MongoDB database. 
- Generating Tokens: Upon successful login, the server generates an encrypted JWT token using a secret key (JWT_SECRET) which encodes the user's ID.
- Frontend Token Storage: The React frontend caches this JWT heavily using Browser localStorage (saved under the key "se_session"). Every subsequent API call attaches this token in the 'Authorization: Bearer <token>' Header. 
- Admin Security: The backend restricts Admin capabilities through a specially verified .env parameter known as the ADMIN_SECRET.

3. KEY FEATURES
- User Dashboard: Users can manage expenses, track daily quantitative habits, and view real-time graphical analytics regarding their monthly spending. 
- Smart Alerts Engine: Custom logical loop that warns the user if spending exceeds 90% of their budget, or if sleep/study averages drop below healthy thresholds.
- Admin Panel: A global overview showing aggregate registered users and overall transactions safely using the ADMIN_SECRET.

PART 2: VIVA QUESTIONS & EXPECTED ANSWERS
-------------------------------------------------------------

Question 1: "Explain how your backend works."
Answer:
"Sir, my backend is built using Node.js and Express.js, and it runs on port 5000. It acts as the bridge between my Next.js frontend and the database. I organized it using a modular REST API structure. Whenever the frontend needs data, it sends an HTTP request (like a GET or POST request) to specific routes—such as '/api/expenses' or '/api/habits'. Once the Express server receives this request, it uses route controllers to run logic, connects to MongoDB Atlas using the 'mongoose' library to save or fetch the data, and then sends that data back to the frontend in a JSON format."

Question 2: "How does the authentication work in your project, and what exactly did you use for it?"
Answer:
"For authentication, I used JSON Web Tokens (JWT). Here is how it works: When a new user registers, I don't save their password directly in plain text. I use a library called 'bcryptjs' to encrypt (hash) their password before storing it securely in MongoDB. When they log in with the correct credentials, my backend generates a secret JWT token and sends it to the frontend. The frontend saves this token in local browser storage. For any subsequent actions—like adding an expense or logging a habit—the frontend sends this token in the request header. My backend's middleware reads the token, verifies the user is authentic, and then permits the action. This makes the system 'stateless' and highly secure."

Question 3: "Tell me the basics about your frontend built here."
Answer:
"The frontend is built using Next.js, which is a powerful React framework, running on port 3000. Instead of basic HTML/CSS, the UI is completely component-based using React hooks like 'useState' to manage live form data and 'useEffect' to fetch data automatically from the backend when pages load. For the design, I wrote custom CSS to make it fully responsive and implemented a dynamic Dark/Light mode toggle. A standout feature of the frontend is that I did not use any bulky external charting libraries. All the analytics—like the Donut pie charts and Bar charts—were created mathematically from scratch using native <svg> components in React, which makes the dashboard load incredibly fast."

Question 4: "Explain the Admin panel. What does it do and how is it secured?"
Answer:
"The Admin panel is a dedicated section built for administrators to monitor the entire platform's activity. It displays global statistics, such as the total number of registered users and the total volume of expenses logged across the whole application. To secure it, I didn't want normal users to arbitrarily access it. So, I implemented an environment variable called 'ADMIN_SECRET' in the backend server code. When an admin logs into the panel, they must provide this secret key. The backend cross-checks this key, and only if it matches does it return the global analytical data. This guarantees user data privacy."

Question 5: "Why use MongoDB instead of MySQL for this project?"
Answer:
"An expense and habit tracker often requires flexible data structures. MongoDB handles unstructured and dynamic JSON-like data (BSON). Furthermore, the mongoose ORM makes it incredibly fast to link my Node.js backend straight to the database."

Question 6: "What is CORS and why did you need it?"
Answer:
"CORS stands for Cross-Origin Resource Sharing. Because my Next.js frontend is running on Port 3000 and my Node.js backend is running on Port 5000, the browser's default security automatically blocks them from talking directly to each other. I used the 'cors' middleware in Express.js to safely allow Port 3000 to fetch data from Port 5000."
